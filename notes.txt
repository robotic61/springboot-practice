2) configure the application.properties to include the database details
spring.jpa.hibernate.ddl-auto=update
Spring Boot checks your Entity.
If table does not exist, create it.
If fields change, update table.

spring.jpa.show-sql=true -> prints SQL queries in console
spring.jpa.properties.hibernate.format_sql=true -> make it readable

3) Add MySQL driver in pom.xml:

JPA = brain 🧠 (knows how to talk to DB)
Driver = cable 🔌 (actual connection)

<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>

6. Custom sql query

Method naming = always gives full entity

Basic structure
Prefix + By + Field(s) + Condition(s) + (OrderBy...)

Example:
findByNameContaining(String keyword)

Field rules
✔ Must match entity field name exactly
✔ Use CamelCase

Method uses PascalCase (Name, Email)
Entity uses camelCase (name, email)

Example:
private String email;
findByEmail(...)

Example:
private String firstName;
findByFirstName(...)

Common prefixes:
findBy     → get data
getBy      → same as findBy
readBy     → same
countBy    → return count
existsBy   → return true/false
deleteBy   → delete rows



✔ You define method name in repository interface
✔ Spring reads the method name
✔ Spring generates a query (JPQL/SQL)
✔ Spring executes it automatically

List<Customer> findByNameContaining(String keyword);

Spring parses method:
findBy + Name + Containing

Spring builds JPQL(object-based query):
SELECT c FROM Customer c WHERE c.name LIKE %:keyword%
Since here Customer is still Java class(not table name).
and c.name to access the object name.

→ Hibernate converts JPQL → SQL

SELECT * FROM customer WHERE name LIKE '%mm%';

Method name → JPQL → SQL → result

✔ Method name MUST match entity fields
✔ Parameter name does NOT matter
✔ Parameter type MUST match the field type
✔ Parameter order matters

My Repository will now have the method

If the method name feels like a sentence → OK(use method naming)
If it feels like a paragraph → use @Query
or database specific features use @Query


Method naming = 80%
@Query = 20%

Switch to @Query when:
✔ Method name is too long
✔ Logic is complex
✔ You need joins / aggregation
✔ You want specific fields only
✔ You need full control over SQL


✔ Yes — List<Customer> always works
✔ It is the safest default if you are unsure

List<Customer> findByEmail(String email);

0 results → empty list ✔
1 result  → list with 1 item ✔
many      → list with many ✔
